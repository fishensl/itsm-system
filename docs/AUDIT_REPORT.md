## 二、关键问题诊断

### 2.1 🔴 业务关联缺失（最严重）

**当前数据状态**：
```
客户未归属地区: 1 / 1  → 100% 客户没填地区
设备未归属地区: 2 / 2  → 100% 设备没填地区
巡检记录关联任务: 0 / 1 → 全部独立巡检，没走任务流程
知识库关联工单: 0 / 0  → 闭环未建立
Fault 与 Ticket 重复: 1 vs 2 → 数据模型重复
```

**问题本质**：
- 模型设计了关联字段（`region_id`, `task_id`, `related_ticket_id`），但**前端表单没有让用户填**
- 地区树设计了但**没在客户/设备表单上选择**
- 巡检任务做了但**和巡检记录没真正打通**
- `Fault` 表是历史遗留，`Ticket` 是新的，**同时存在没合并**

### 2.2 🔴 Fault 与 Ticket 模型重复

```
旧表 faults    →  历史故障记录（保留）
新表 tickets   →  工单（含派单流程）
```

**两套并存的后果**：
- 工作台 4 个统计卡：`故障` + `工单` 含义重叠
- 用户报修一次，分不清填哪个
- 知识库关联两套 ID

**建议**：
- 把 `Fault` 标记为「已迁移」入口，新业务全部走 Ticket
- 工作台合并 "故障" 卡片
- 旧故障记录前端入口保留只读

### 2.3 🟡 工单流程虚有其表

**当前 2 条工单全部停留在"待派单"状态**，说明：
- 没人真正点过"派单 → 接单 → 处理 → 审核 → 验收"
- 缺少**默认派单逻辑**（应该派给当前用户或值班人员）
- 缺少**待办提醒**（创建工单后接收人怎么知道？）

### 2.4 🟡 巡检模块三件套断裂

设计的链条：
```
巡检模板 → 巡检任务 → 巡检记录
```

实际状态：
```
0 模板  →  0 任务  →  1 独立记录
```

**问题**：
- 模板表存在，但创建任务/记录时没强制选模板
- 任务创建后没有"执行任务"按钮跳转到巡检表单
- 巡检任务页缺少看板视图（参考图有）

### 2.5 🟡 设备多凭证/接口模型未启用

```
DeviceCredential 表 → 0 条
DeviceInterface 表 → 0 条
```

设备表自身有 `username` / `password_encrypted` / `interface`，但同时建了多凭证/多接口表 — **存在数据冗余风险**。

**建议**：
- 短期：设备表内字段作为「主凭证」
- 长期：迁移到 DeviceCredential 多对一关系

### 2.6 🟡 权限系统未落地

```
权限管理页面: 桩路由（"权限管理页面 (S10)"）
```

但用户管理已经有 `admin/operator` 两个角色。差距：
- 缺少 **角色定义表** 和 **权限矩阵**
- 现在所有 operator 都能改所有数据
- 计划中的 Role/Permission 模型还没建

### 2.7 🟡 数据报表完全是桩

```python
@app.route('/dashboard/reports')
def dashboard_reports():
    return '数据报表页面 (S13)'
```

工作台底部那个饼图是唯一的可视化，没有真正的报表模块。

### 2.8 🟡 模板 19 个文件仍有 `fas fa-*`

旧 FontAwesome 没替换干净，导致部分图标显示空白。涉及：
```
topologies, tickets/*, knowledge_base/form,
inspectors, inspections/*, faults/*, devices/*,
ai_config/*, reports
```

### 2.9 🟡 app.py 单文件 2730 行

- 维护困难：找一个路由要翻几屏
- 不能利用 Flask Blueprint 分模块
- 路由 + 视图函数全部在一起，无 service 层

## 四、性能 / 数据库问题

### 4.1 缺少索引
- 32 个表中只有 `password_history.device_id` 是 indexed
- 高频查询字段缺索引：
  - `tickets.customer_id`, `tickets.status`, `tickets.assigned_to`
  - `inspections.customer_id`
  - `devices.customer_id`, `devices.region_id`
  - `device_config_backups.device_id`

### 4.2 工作台 N+1 查询
`index()` 中循环 Customer.query.get() 取客户名，应该用 JOIN。

### 4.3 列表页未全部分页
不少新模块（备件、销售、采购）查询全表后 .all()，未来数据量大时会卡。

## 八、数据迁移建议

**Fault → Ticket 合并方案**：
```sql
-- 把所有未关联 ticket 的 fault 转成 ticket
INSERT INTO tickets (number, title, description, customer_id, ...)
SELECT 'WO-LEGACY-' || id, title, fault_description, customer_id, ...
FROM faults WHERE ticket_id IS NULL;

-- 然后给 faults.ticket_id 回填
```

## 九、模块整合大图

理想关联应该是：

```
┌──────────────────────────────────────────────────┐
│              地区树 (Region)                       │
└──────────────────────────────────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
   客户 (Customer) ─── 设备 (Device) ─┘
        │              │
        │     ┌────────┴──────┐
        │     │               │
        │  巡检任务         配置备份/采集
        │  (InspectionTask)  (Backup/Collect)
        │     │
        │     ▼
        │  巡检记录 (Inspection)
        │     │
        │     │ 发现异常
        │     ▼
        └──→ 工单 (Ticket)
                │ 处理完成
                ▼
            知识库 (Knowledge)
```

**当前断裂的箭头**：
- 地区 → 客户/设备（关联字段没填）
- 巡检任务 → 巡检记录（task_id 没填）
- 工单 → 知识库（缺少一键创建）

## 十、结论

**总体评价**：
- ✅ 框架完整、UI 风格统一（最近压缩了行高）
- ✅ 数据模型设计较完整（32 张表覆盖业务）
- 🔴 **关联逻辑没真正打通**，模块各自为政
- 🔴 部分模块（拓扑图、权限、报表、AI）仅有壳
- 🟡 代码组织需要拆分

**建议下一阶段重点**：
**不再做新模块，把现有模块的"关联链条"打通。** 让每一条业务数据都能从地区出发，关联到客户、设备、巡检、工单、知识库，形成完整闭环。
